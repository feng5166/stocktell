# 《埋点需求》— 数据埋点 + 漏斗看板

> 目标:看到核心增长漏斗 **访问 → 加自选 → 注册 → 绑推送 → 次日回访** 的每一跳转化率,
> 为产品决策(哪一步流失最狠)提供依据。
> 选型:**VPS 自托管 Umami**(轻量、隐私友好、数据不出境)。

---

## 1. 目标与核心漏斗

| 步 | 里程碑 | 含义 | 度量口径(唯一访客/用户) |
|---|---|---|---|
| 1 | 访问 | 打开站点任意页 | Umami 唯一访客(visitor) |
| 2 | 加自选 | 点 ★ 把个股/ETF 加入自选 | 触发过 `add_watchlist` 的唯一访客 |
| 3 | 注册 | 成功创建账号(邮箱/Google) | 触发过 `signup` 的唯一访客 |
| 4 | 绑推送 | 绑微信 或 开启邮件/浏览器推送 | 触发过 `bind_push` 的唯一用户 |
| 5 | 次日回访 | 注册后次日(D1)再次访问 | D1 留存(Umami Retention) |

**每跳转化率** = 本步唯一访客 ÷ 上一步唯一访客。

> ⚠️ 口径说明:真实产品里这 5 步**并非严格时序**(游客可先加自选再注册;Google 注册会跳过弹窗)。
> 因此漏斗按「**唯一访客是否触达过该里程碑**」计,而非"必须按顺序发生在同一会话"。
> 这是更诚实、对小体量更稳的口径;Umami 的 Funnel 报表是会话内时序的,仅作可视化参考,
> 关键转化率以「各里程碑唯一访客比值」为准(见 §6)。

---

## 2. 选型建议:Umami(自托管)

**结论:用 Umami,自托管在现有 VPS。** 理由:

- **轻量**:单 Node 服务 + 一个 Postgres 库,跑在已有 VPS(47.84.8.167)即可,与现有 `ilink-bridge` 同机同模式(systemd + nginx 反代)。
- **隐私友好 / 合规**:默认不写 cookie、不收集 PII,**无需 cookie 同意弹窗**;数据自托管、不出境(符合国内合规直觉)。
- **够用**:支持自定义事件(`umami.track`)、UTM、Referrer、设备/地域;v2 自带 **Funnel(漏斗)** 与 **Retention(留存)** 报表,正好对上需求。
- **零成本**:开源、无量级计费。

**对比(为什么不是别的):**

| 工具 | 漏斗 | 自托管 | 重量 | 结论 |
|---|---|---|---|---|
| **Umami** | ✅ v2 Funnel + Retention | ✅ | 极轻 | ✅ 选它 |
| Plausible | 漏斗需付费版/较弱 | ✅ | 轻 | 漏斗弱,排除 |
| PostHog | ✅ 强(漏斗/留存/回放) | ✅ 但重(ClickHouse+Redis+多容器) | 重 | 体量不匹配,先不上 |
| Matomo | ✅ | ✅ | 重(PHP/MySQL) | 偏重、运维成本高 |
| GA4 | ✅ | ❌(数据出境) | — | 出境+隐私,排除 |

**已知局限(写明,别踩坑):**
- Umami 的 **Funnel 报表是「单会话内、按 URL/事件顺序」** 的;跨天、乱序的产品级漏斗它算不准。
  → 故 §6 用「唯一访客触达各事件」的比值作为权威转化率,Funnel 报表仅作趋势参考。
- **Google 注册**走 OAuth 跳转,弹窗内的 `signup` 不一定能埋到 → 见 §5 的处理方案(首登信号补埋)。
- 「次日回访」用 Umami **Retention** 报表看 D1;若要精确到「注册用户的 D1」,二期可加服务端 `last_active` 兜底(见 §7)。

## 3. 部署架构(VPS 自托管)

```
用户浏览器
   │  GET https://analytics.stocktell.me/script.js   (Umami 采集脚本)
   │  POST .../api/send                               (上报事件)
   ▼
Cloudflare(TLS) ──► VPS nginx 反代 ──► Umami(Node, 127.0.0.1:3001, systemd)
                                           │
                                           ▼
                                     Postgres(umami 库,本机)
```

- **新增子域**:`analytics.stocktell.me`(Aliyun DNS 加 A 记录 → 47.84.8.167;TLS 沿用 `bridge.stocktell.me` 同款,Cloudflare 代理 / 或 certbot)。
- **Umami**:Node 服务,监听本机 3001,`systemd` 守护(仿 `ilink-bridge.service`)。
- **Postgres**:VPS 本机新建 `umami` 库(分析数据与业务库隔离;业务库仍是 Vercel/Neon,不混)。
- **nginx**:新增 server 块 `analytics.stocktell.me` → `proxy_pass http://127.0.0.1:3001`。
- 初始化:`npx umami`(或源码部署),建管理员账号,新建 website 拿到 `website-id`。

## 4. 接入方式(前端)

1. **采集脚本**:在 `src/app/layout.tsx` 的 `<head>` 注入(仅生产环境):
   ```html
   <script defer src="https://analytics.stocktell.me/script.js"
           data-website-id="<WEBSITE_ID>"></script>
   ```
   → 自动采集 **pageview(访问)**,SPA 路由切换也会自动上报。

2. **事件封装** `src/lib/analytics.ts`(安全包一层,Umami 没加载/被拦截也不报错):
   ```ts
   type Props = Record<string, string | number | boolean>;
   export function track(event: string, props?: Props) {
     if (typeof window === "undefined") return;
     try { (window as any).umami?.track?.(event, props); } catch {}
   }
   ```
   各埋点处 `import { track } from "@/lib/analytics"` 调用。

## 5. 事件清单(埋点表)

> 命名:小写下划线;属性尽量精简(Umami 单事件属性别堆太多)。**不上报任何 PII**(不传邮箱/手机号)。

| 事件名 | 触发时机 | 代码位置(建议) | 属性 | 属于漏斗 |
|---|---|---|---|---|
| *(pageview)* | 任意页加载/路由切换 | Umami 脚本自动 | 自动(url/referrer/UTM) | ① 访问 |
| `add_watchlist` | 点 ★ **加入**自选成功 | `components/useWatchlist.ts` 的 `toggle`(adding 分支) | `kind`(stock/etf)、`source`(当前路径) | ② 加自选 |
| `remove_watchlist` | 取消自选 | 同上(删除分支) | `kind` | — |
| `signup` | **新账号**创建成功 | 邮箱:`AuthModal` 注册成功后;Google:见下注 | `method`(email/google) | ③ 注册 |
| `login` | 老用户登录成功 | `AuthModal` signIn 成功 | `method` | — |
| `bind_push` | 绑微信 / 开启邮件或浏览器推送成功 | `PwaActions`(开微信/浏览器推送)、`/settings`(邮件开) | `channel`(weixin/web/email) | ④ 绑推送 |
| `view_briefing` | 进首页看简报 | `app/page.tsx`(或首屏组件 mount) | — | 辅助 |
| `view_stock` | 进个股详情页 | `app/stock/[code]`(客户端组件 mount) | `code` | 辅助 |
| `deep_read` | 点「让 StockTell 深读」 | `components/DeepRead.tsx` / `StockTellTake` | `kind`(stock/morning/fundflow) | 辅助(高意向) |
| `submit_feedback` | 提交反馈成功 | `components/FeedbackLink.tsx` | `category` | 辅助 |
| `vote_chain` | 「我想要」其他产业链投票 | `components/ChainSwitcher` | `chain` | 辅助 |
| `overnight_radar_view` | 首页跨市场预期差雷达出现信号(本次挂载首次) | `components/OvernightRadar.tsx` | `signals`(信号数) | 美股桥·曝光 |
| `overnight_radar_click` | 雷达卡点对应 A 股 → 直达个股(核心"发现→行动") | `components/OvernightRadar.tsx` | `us`、`a`、`strength`、`linkage`(有效率 0–1,无则 -1) | 美股桥·转化 |
| `predict_diff_click` | 主动发现(预期差)tab 点对应 A 股 | `components/Dashboard.tsx` 的 `ActiveDiscovery` | `us`、`a`、`strength` | 美股桥·转化(次) |
| `relation_strong_only` | 关联图谱开「仅看强关联」 | `components/Dashboard.tsx` 的 `RelationMap` | — | 美股桥·参与 |

> **美股桥漏斗**(跨市场桥接转化率 = 北极星):`overnight_radar_view → overnight_radar_click → add_watchlist`。
> 用 Umami 访客重叠看:看到雷达的访客里,多少点了对应 A 股、又有多少加了自选。`linkage`/`strength` 属性可下钻"强关联/高有效率是否更易转化"。

**`signup` 的 Google 处理**:OAuth 跳走后回站,弹窗代码不一定执行到。方案二选一:
- (推荐)在 NextAuth `signIn` 回调里,**首次创建用户**时给前端一个信号(如 `?welcome=1` 跳转参数),首页据此补埋 `signup{method:'google'}`;
- 或接受 Google 注册数由「服务端 users.createdAt」兜底统计(漏斗里 Google 注册走 DB 校准)。

## 6. 漏斗 & 留存定义(权威口径)

**漏斗(取同一时间窗,如近 7/30 天):**

| 跳转 | 转化率公式 |
|---|---|
| 访问 → 加自选 | `uv(add_watchlist) / uv(pageview)` |
| 加自选 → 注册 | `uv(signup) / uv(add_watchlist)` |
| 注册 → 绑推送 | `uu(bind_push) / uu(signup)` |
| 绑推送 → 次日回访 | `D1留存(绑推送当日活跃用户)` |

> `uv`=唯一访客,`uu`=唯一(登录)用户。
> 在 Umami:用 **Events 报表**取每个事件的「唯一访客数」算前 3 跳;用 **Funnel 报表**做可视化趋势;
> 用 **Retention 报表**看「次日(Day 1)回访率」。

**次日回访(D1):** 当日有活跃(访问/任意事件)的访客中,**次日再次出现**的比例。Umami Retention 直接出。
若要「注册用户的 D1」更精确,见 §7 二期。

**交付即「能看到」:** Umami 后台一个 Dashboard,4 个卡片/报表拼出上面 4 跳 + D1,口径如上。

## 7. 实施计划(分阶段)

**P0 — 部署 Umami(运维,~半天)**
- VPS 装 Node + Postgres(本机),部署 Umami,systemd 守护;
- DNS 加 `analytics.stocktell.me`,nginx 反代,TLS;
- 建管理员 + website,拿 `WEBSITE_ID`。

**P1 — 接入埋点(研发,~半天)**
- `layout.tsx` 注入采集脚本(仅生产,用环境变量 `NEXT_PUBLIC_UMAMI_*` 控制);
- 加 `lib/analytics.ts`;
- 按 §5 在 5 个核心点 + 辅助点埋 `track(...)`;
- 验收:Umami 实时看到 pageview + 各事件进来。

**P2 — 配置看板 + 校准(~1 小时)**
- Umami 建 Funnel(访问→加自选→注册→绑推送)+ Retention(D1);
- 与 DB 交叉校准注册/绑推送数(尤其 Google 注册);
- (可选)给 `users` 加 `last_active_at`,服务端兜底精确「注册用户 D1」。

**回归**:埋点是「触发即上报」,不影响主流程(`track` 包了 try/catch);`scripts/smoke.sh` 可加一条「analytics.stocktell.me/script.js 200」可达性检查。

## 8. 隐私 / 合规
- Umami 默认**不写 cookie、不收集 PII**;埋点属性**严禁**传邮箱/手机号/姓名,只传 code/kind/method 这类非身份信息。
- 数据自托管在自有 VPS,**不出境**、不进第三方。
- 无需 cookie 同意弹窗(因不依赖 cookie 跟踪)。

## 9. 验收标准
- Umami 后台能看到 **访问 → 加自选 → 注册 → 绑推送** 4 跳的**每跳转化率**,以及**次日回访(D1)**;
- 数据每日自然累积,可按日期范围 / 来源(UTM/referrer)拆分;
- 埋点不影响线上主流程(失败静默)。
