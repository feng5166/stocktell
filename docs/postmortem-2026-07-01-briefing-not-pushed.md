## 4. 根因分析
**直接原因(最可能):07:00 的 cron 根本没触发。**
判据:若触发,必走某分支并发飞书告警(0条/休市/报错都会),但**既无简报又无任何告警**;而数据当时是好的(dryRun 8 条)、cron 已注册、账号已 Pro。最合理解释 = **Hobby→Pro 过渡期,该时段 cron 未在新计划下生效**(Vercel 升级 + 重新部署后 cron 才在 Pro 下重新注册)。

**保留不确定**:Vercel 运行时日志经 API 拿不到(只给构建日志),无法 100% 证明"没触发";且飞书投递本身当时有静默失败史,"没收到告警"非铁证。但"无简报 + 无告警 + 数据好"三者合起来,指向"没触发"最强。

**放大问题的系统缺陷(为什么小事拖成一天)**
1. **无看门狗**:主 cron 静默失败后,数小时无人知,直到用户发现。
2. **告警依赖 cron 自身触发**:cron 没跑 → 没有任何信号。监控存在盲区。

**排查中连带暴露的独立隐患(顺手一并修了)**
- 飞书 `sendFeishu` 静默吞错 → `pushed:8` 假成功、真失败不可见。

## 10. 经验教训
1. **关键定时任务必须有独立看门狗**:别让告警依赖"任务自己跑起来"。
2. **静默吞错是隐形炸弹**:sendFeishu/sendMail 之前都"失败也返回成功",要让失败可见、可记录。
3. **批量外呼要节流+重试**:第三方(Resend/飞书/Tushare)都有限流,紧循环必偶发丢。
4. **改计划(Hobby→Pro)/大改部署后,主动验证 cron 与定时链路**,别默认沿用。
